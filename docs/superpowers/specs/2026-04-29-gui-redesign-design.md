# DeckGen GUI Redesign — Design Spec

**Date:** 2026-04-29
**Status:** Approved for implementation

---

## Goal

Replace the current `gui.py` (single-page, sidebar-heavy, confusing) with a clean
3-tab browser GUI that guides users from dataset selection through deck generation
and validation. The new GUI eliminates all sidebar clutter, exposes collateral-driven
discovery, and makes each generated deck immediately inspectable.

---

## Architecture Overview

```
gui.py  (Flask/http.server, single file, serves HTML + JSON API)
  |
  +-- GET  /                     → serves redesigned HTML shell
  +-- GET  /api/nodes            → list available nodes
  +-- GET  /api/libtypes?node=   → list lib types for a node
  +-- GET  /api/corners?node=&libtype=  → list corners (from manifest)
  +-- GET  /api/cells?node=&libtype=   → cell list with arc counts
  +-- GET  /api/arcs?node=&libtype=&cell=  → arc list + index_1/index_2 for a cell
  +-- POST /api/generate         → run batch job, stream progress (SSE/chunked)
  +-- GET  /api/deck?path=       → return raw SPICE text for a deck
  +-- POST /api/rescan?node=&libtype=  → run scan_collateral.py, return updated manifest
```

The frontend is a **single HTML page** served by the existing Python HTTP server.
No JS framework — vanilla JS only, matching the current `gui.py` pattern.
The collateral data source is always `manifest.json` (generated by `scan_collateral.py`).

`/api/generate` streams progress via **chunked transfer encoding** (not full SSE).
The server already uses `http.server`; chunked responses are added by flushing
`sys.stdout` inside the handler and setting `Transfer-Encoding: chunked`.
Each chunk is a newline-delimited JSON object.

`/api/arcs` response shape:
```json
{
  "arcs": [
    {
      "arc_id": "hold_CELL_QN_rise_CP_rise_NO_CONDITION",
      "arc_type": "hold",
      "probe_pin": "QN", "probe_dir": "rise",
      "rel_pin": "CP",   "rel_dir":   "rise",
      "when": "NO_CONDITION",
      "index_1": [0.1, 0.2, 0.5, 1.0, 2.0],
      "index_2": [0.05, 0.1, 0.2, 0.5, 1.0]
    }
  ]
}
```
`index_1`/`index_2` are the raw float arrays from `template.tcl`; the Sweep button
uses these to populate `(i,j)` pairs for all combinations.

Table-point text input format: `(i1,i2)` pairs separated by whitespace,
e.g. `(1,1) (2,3) (4,4)`. Backend parses with regex `\((\d+),\s*(\d+)\)`;
invalid tokens are skipped with a warning logged.

## Tab 1 — Explore

**Purpose:** User does not know exactly what to simulate. Browses cells and arcs
from the loaded dataset, adds individual arcs to a queue, selects table points,
then generates.

### Layout: 2-column grid (`1fr 380px`)

#### Left panel — Cells & Arcs

- **Search bar** — filters cell list by substring (client-side)
- **Arc-type filter chips** — All / hold / setup / combinational / recovery / removal / mpw
  - Clicking a chip filters the arc list inside each expanded cell
- **Cell list** — each cell row has:
  - Collapse/expand twisty (►/▼)
  - Cell name (monospace)
  - Arc-count tags per type (e.g. `hold:4 setup:4 mpw:2`)
  - Clicking the row header toggles the arc sub-list
- **Arc sub-list** (inside expanded cell):
  - Each row: arc-type tag + `probe_pin/dir · rel_pin/dir · when_condition`
  - "+ Add" button on hover → adds arc to queue; button becomes "✓ added" and row dims
  - Arc rows that are already in the queue stay dimmed with green "added" state
- Data source: `/api/cells` and `/api/arcs`

#### Right panel — Queue

**State A: Building queue (before Generate)**

1. **Selected arcs section**
   - Label row "Selected arcs" + "Clear all" ghost button
   - One row per queued arc: `[type-tag]  cell | probe/dir · rel/dir  [×]`
   - × removes that arc from the queue
   - Empty state: italic placeholder text

2. **Table points per arc-type section**
   - Label "Table points per arc-type"
   - Hint: "Enter (i1,i2) pairs — e.g. `(1,1) (2,3) (4,4)`"
   - One input row per arc-type **present in the queue** (no rows for absent types):
     - `[type-tag]  [text input: "(1,1) (3,3) (4,4)"]  [Sweep]`
   - **Sweep**: populates the input with all (i1,i2) pairs from the template.tcl
     `index_1 × index_2` for that arc-type (fetched from `/api/arcs`)
   - Input is free-text; parsed on the backend at generate time

3. **Summary section**
   - One line per arc-type: `[type-tag]  N arcs × M pts  =  K decks`
   - Total line: `K total decks × C corners  =  N final`
   - Updated live as user edits inputs

4. **Footer actions**: `[Preview]  [Generate N decks ▶]`

**State B: Results (after Generate)**

- Panel header shows "40 decks generated"
- "← Back to queue" button restores State A
- Results list: one row per (arc × corner) deck:
  - Green/red status dot + arc id (truncated) + corner name
  - Clicking any row opens **Deck Viewer**
- Footer: `[Copy all paths]  [Open output dir]`

#### Deck Viewer

- Overlays the **left panel** only (right panel stays visible with results list)
- Fixed position: `top: dataset-bar-bottom; left: 0; right: 380px; bottom: 0`
- Header: "Deck" label + full arc_id · corner + `[× Close]  [Copy]`
- Body: dark-background `<pre>` with syntax highlighting (comments, keywords,
  params, numbers — applied via regex on the raw SPICE text)
- Content fetched from `/api/deck?path=<output_path>`
- Closing via × or switching tabs dismisses overlay

## Tab 2 — Direct

**Purpose:** User has a list of `cell_arc_pt` identifiers already (from MCQC logs,
spreadsheets, etc.) and just wants to generate decks without browsing.

### Layout: 2-column grid (`1fr 1fr`)

#### Left — Identifier input

- Full-height `<textarea>` (monospace), one identifier per line
- Header: "cell_arc_pt identifiers" + `[Load file…]  [Clear]`
- `[Load file…]` opens a file picker; contents replace textarea
- Textarea is parsed live (debounced 300 ms)

#### Right — Parsed summary

- Header: "Parsed summary" + count pill `"6 arcs × 4 corners = 24 decks"`
- Arc-type rows: one per type detected, with arc count
- Note: i1/i2 are taken from identifier suffix `_N_M` — no table-point selection
- Validation errors shown inline (red) for malformed lines
- Footer: `[Preview]  [Generate N decks ▶]`

---

## Tab 3 — Validate

**Purpose:** Compare a DeckGen output directory against an MCQC reference directory
using the existing `validate_decks.py` logic.

### Layout: Full-width single panel

- **Path inputs**: DeckGen output root, MCQC output root, file selector (nominal/mc)
- **Stat cards** (4): Total pairs / L1 identical / L2 normalized match / L3 different
- **Filter bar**: All / L3 only / hold / combinational / mpw / …
- **Results table**: type | arc id | level (L1/L2/L3) | top diff class | lines diff | [View diff]
- `[View diff]` opens a side-by-side diff overlay (future; button shown, not wired)
- `[Run validation]` calls `POST /api/validate` which invokes `validate_decks.py`
  as a subprocess and returns the JSON summary + pair list
- `[Export HTML report]` calls `GET /api/validate/report` to download the HTML file

## Dataset Bar (shared, Explore + Direct only)

| Control | Behaviour |
|---------|-----------|
| Node dropdown | Populated from `/api/nodes`; changing resets lib type + corners |
| Library type dropdown | Populated from `/api/libtypes?node=`; changing resets corners |
| Corners chip-picker | Dropdown with search + checkboxes; selected corners shown as chips |
| Rescan button | Calls `scan_collateral.py` for current node, refreshes manifest |

Status pill in topbar updates to reflect: `{node} / {libtype_short} / {n} corners / {cells} cells`

---

## Corners Chip-Picker

- Trigger element looks like a `<select>` but shows selected corners as small chips
- Dropdown: sticky search input + scrollable checkbox list
- Selecting/deselecting a corner immediately updates the status pill and queue summary
- Closes on outside click

## Generate Flow (backend)

```
POST /api/generate
Body: {
  mode: "explore" | "direct",
  node, libtype,
  corners: ["ssgnp_0p450v_m40c", ...],
  arcs: [                          // explore mode
    { arc_id: "hold_..._1_1", ... }
  ],
  identifiers: ["hold_..._4_4", ...],  // direct mode
  output_dir: "./output/"
}

Response: SSE stream
  data: {"status": "progress", "done": 3, "total": 40, "current": "hold_..._ssgnp_..."}
  data: {"status": "done", "succeeded": 38, "failed": 2, "results": [...]}
```

- Backend parses table-point strings `"(1,1) (3,3)"` into `[(1,1), (3,3)]`
- Calls existing `core/batch.py` `run_batch()` per arc × corner
- Output layout: `{output_dir}/{libtype}/{corner}/{arc_type}/{arc_id}/nominal_sim.sp`
- Each result record: `{arc_id, corner, status, path, error}`

## What is Removed

The following sections from the current `gui.py` are **deleted entirely**:

- Left sidebar: Inputs, Corners, Files, Overrides panels
- Single-mode arc form (all fields)
- Validation/status sidebar panel
- "Collateral Mode" toggle

The new GUI **always** operates from the collateral dataset. Direct CLI usage remains
for users who prefer it.

---

## Error States

| Situation | UI behaviour |
|-----------|-------------|
| No manifest found for node/libtype | Dataset bar shows warning; Explore panel shows "Run Rescan to build manifest" |
| Arc added but no template match | Queue row shows ⚠ icon; deck shows error text instead of SPICE |
| Generate fails for a deck | Result row shows red dot + error message; deck viewer shows error |
| Malformed identifier (Direct) | Red underline on that line + tooltip |

## Non-Goals (out of scope for this implementation)

- FMC / MC run integration (future)
- Side-by-side diff viewer in Validate tab (button shown, not wired)
- Multi-node batch (one node/libtype per session)
- Authentication or multi-user support
